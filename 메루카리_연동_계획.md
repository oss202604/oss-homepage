# 메루카리 연동 — 최종 실행계획 (자가치유 설계, 멀티에이전트 2회차 2026-06-23)

> 아키텍트3→크리틱→플래너(7에이전트) 2회. 1회차: 상시폴링=월₩300만 적발→온디맨드. 2회차(자가치유): 사장님 숙제(불안정→안정·Claude상주·99%) 해결 + **크리틱이 "자체 스크래퍼 주력"을 정정**.

## ⭐ 핵심 업데이트 — 사장님 일본 거주 (게임체인저, 2026-06-23)
**사장님이 일본 거주 + 일본에서 운영** → 자체 스크래퍼의 최대 약점이던 "일본 IP 프록시 비용"이 소멸:
- 일본 VPS(Sakura/ConoHa ~¥600~2000/월) 또는 일본 인터넷 = **native 일본 IP → 프록시 0원**. 비일본 403 문제 해소.
- → **자체 스크래퍼(재팬큐식)가 현실적 주력으로 부활**, Apify는 폴백으로 강등. (1·2회차 분석의 "Apify 주력"은 '일본IP=유료프록시' 전제였는데 그 전제가 깨짐.)
- ⚠️ IP와 별개로 남는 과제(정직): Cloudflare 봇보호·DPoP 처리 필요 / 일본 VPS도 데이터센터 IP라 일부 챌린지 가능(집 residential IP가 최선이나 24h 운영엔 불안정) / RSC·유지보수는 그대로. → **Phase 0에서 일본 IP로 실제 안 막히는지 검증 후 확정.**
- 비용 재추정: 자체(일본 VPS) 주력이면 **월 ₩3~8만**(VPS ₩수천~2만 + 번역 ₩1만↓ + Supabase, Apify는 폴백이라 평소 0)으로 Apify주력(₩7~16만)보다 저렴.
- ※ 자가치유·stale serving·Claude상주·99% 화면가용성 설계는 자체든 Apify든 **동일하게 적용**(공급선만 바뀜).

## 사장님 숙제 → 해결책
- **불안정→안정**: **긁는 곳(공급)과 보여주는 곳(Supabase+Netlify) 물리 분리.** 진실원천=Postgres mirror DB. **다단 폴백**: L1 모바일API(DPoP) → L2 Playwright → L3 정적HTML → Apify → **stale serving**(다 죽어도 DB 캐시 그대로 노출 = **빈화면 0**). 5단계 우아한 강등(L0정상~L4 카탈로그만 숨김, 본업 주문/배송은 항상 정상).
- **Claude를 상주 개발자로**: pg_cron이 5~15분마다 헬스체크 → 이상 시 **Claude 자동조치**(method_priority 전환·Apify 비상ON·scraper_config 갱신·서킷리셋) → **자동으로 못 고치면**(Cloudflare 근본변경·CAPTCHA·RSC변경) Slack #oss-시스템경보로 "①Claude가 한 일 ②손님 영향 ③사장님 1액션" 3줄 보고. 코드수정은 **PR 제안→사장님 1클릭 머지**(무검수 자동배포 금지).
- **99% (정직하게 2개로 분리)**: **카탈로그 화면 가용성 99.9%**(stale-serving 덕, 약속가능·세일즈포인트) / **가격 최신연동 95~98%**(메루카리 협조 의존, 99% 보장 불가·정직 고지). "막히면 자동 Apify·점검중 강등 + 손님이 살 때 그 상품만 최신확인."

## 실시간 동기화
공식 웹훅 없음→진짜 실시간 불가. 3층: (A)상세 **클릭 순간** 그 상품만 라이브 재조회(가격·sold) (B)그리드=증분크롤 분~시간 (C)신착띠=인기 사전캐시. pgmq 큐 + source_hash 변경감지 + price_history. cron 하드캡+일일 비용가드(상시 풀폴링=월폭탄 금지).

## 비용 / 가용성
- 평시 **월 ₩7~16만** (Apify spend-limit $30 + Google번역 ₩1만↓ + Supabase). 자체VPS는 후순위(프록시필수라 비용추가).
- 풀카탈로그 첫날 전량미러 금지(폭탄) → **인기+조회분 lazy mirror**로 시작, 매출 받쳐주면 확대.

- **Phase 0 (mvpFirstStep·₩0)**: 사장님 Apify 무료가입(이메일만·카드X)→토큰→Claude가 무료 2000건으로 액터 A·B 2개 테스트(필드·사진403/200·실비용)→1장 보고. 되면 진행.
